# قوانين وإرشادات مشروع تطبيق تأجير الوحدات الساحلية في مصر

## 1. أسلوب العمل وإدارة المهام (Workflow)
- المشروع يدار بأسلوب التطوير التدريجي الصارم (Incremental Development).
- يتم تقسيم العمل إلى مهام متناهية الصغر (Granular Tasks).
- يُمنع منعاً باتاً الانتقال من Task إلى أخرى إلا بعد مراجعة الـ Product Owner واعتمادها بالكامل.
- الترتيب الإلزامي: (تحليل وشرح الـ Task -> موافقة الـ Product Owner -> كتابة كود الـ Task فقط -> تقديم تقرير الإنجاز -> الاعتماد النهائي).

## 2. القيود البرمجية والمعمارية (Architecture & Code Quality)
- **البنية المعمارية:** الالتزام الصارم بـ Clean Architecture وفصل المكونات (Separation of Concerns).
- **التصميم:** الالتزام بـ Modular Design لضمان إمكانية التوسع (Scalability) وسهولة الصيانة (Maintainability).
- **أفضل الممارسات:** تطبيق مبادئ SOLID وقواعد الكود النظيف (Clean Code).
- **الحدود:** لا يُسمح بكتابة أي كود لم يُطلب صراحةً، ولا تُنشأ ملفات خارج نطاق المهمة الحالية، ولا تُعدل ملفات خارج نطاق الـ Task.
- **القرارات التصميمية:** أي قرار يخص قاعدة البيانات، التصميم، أو اختيار المكتبات يجب أن يُعرض على الـ Product Owner للموافقة أولاً.

## 3. معايير التقارير (Post-Task Report)
بعد انتهاء كل Task، يجب تقديم تقرير مفصل يحتوي على:
1. ماذا تم إنجازه بدقة؟
2. ما هي الملفات التي أُنشئت؟
3. ما هي الملفات التي تم تعديلها؟
4. الأسباب التقنية وراء هذه التعديلات والإنشاءات.
5. ما هي الخطوة التالية المباشرة؟
